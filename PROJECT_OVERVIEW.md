# Room Booking Service — Project Overview

## Что это
Сервис бронирования переговорок на Go для тестового задания.

Сервис должен позволять:
- администраторам создавать переговорки;
- администраторам один раз задавать расписание доступности переговорки;
- системе генерировать 30-минутные слоты с устойчивыми UUID;
- пользователям смотреть доступные слоты;
- пользователям создавать и отменять свои брони;
- администраторам смотреть все брони с пагинацией;
- пользователям смотреть только свои будущие брони.

## Основные требования
- язык реализации: Go;
- база данных: PostgreSQL;
- запуск через `docker-compose up --build`;
- `docker-compose.yaml` в корне репозитория;
- сервис слушает порт `8080`;
- `GET /_info` всегда возвращает `200 OK`;
- API реализуется по `docs/assignment/api.yaml`.

## Архитектурный курс
Проект делается как один Go backend-service без микросервисной сложности.

Опорные принципы:
- OpenAPI-first;
- PostgreSQL как единственная БД;
- materialized slots с устойчивыми UUID;
- UTC-only для дат и времени;
- бизнес-правила оформлены в domain/application layer;
- консистентность бронирования удерживается транзакциями и ограничениями БД.

## Доменная модель
Базовые сущности:
- `User`
- `Room`
- `Schedule`
- `Slot`
- `Booking`

### Room
Переговорка с названием, описанием и вместимостью.

### Schedule
Одно правило доступности для переговорки:
- `daysOfWeek`
- `startTime`
- `endTime`

После создания не редактируется.

### Slot
Материализованный 30-минутный интервал в UTC:
- имеет стабильный UUID;
- принадлежит комнате;
- создаётся системой на основе расписания.

### Booking
Связь пользователя со слотом:
- `active`
- `cancelled`

Отмена идемпотентна.

## Ключевое архитектурное решение: как генерируются слоты
Для этого проекта разумнее хранить слоты в базе, а не генерировать их заново на каждый запрос.

Почему:
- по заданию `slotId` должен быть стабильным;
- самый нагруженный endpoint — список доступных слотов по комнате и дате;
- бронирование идёт по `slotId`, значит слот должен существовать как запись;
- проще держать уникальность, тестируемость и предсказуемое поведение.

Базовая стратегия:
1. администратор создаёт комнату;
2. администратор создаёт расписание один раз;
3. система генерирует слоты на rolling window вперёд;
4. endpoint списка слотов читает уже подготовленные слоты из БД;
5. бронь помечает слот занятым через активную booking-запись.

## Бизнес-инварианты
- один слот может быть занят только одной активной бронью;
- слоты одной переговорки не должны пересекаться;
- без расписания слоты не создаются;
- нельзя бронировать слот в прошлом;
- admin не может создавать бронь;
- user не может отменять чужую бронь;
- повторная отмена уже отменённой брони возвращает `200`;
- `/bookings/my` возвращает только будущие брони.

## Технические риски, которые надо закрыть реализацией
- дубли слотов при повторной генерации;
- race condition при одновременном бронировании;
- рассинхрон с `api.yaml`;
- ошибки работы со временем и UTC;
- смешение бизнес-логики между handler и SQL.
